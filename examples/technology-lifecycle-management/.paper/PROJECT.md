# Technology Lifecycle Management Example

Keep this file short. `PROJECT.md` captures project identity and operating constraints; `BRIEF.md` owns the detailed thesis, claims, objections, and definition of done.

## What This Is

This is an anonymized internal strategy / decision memo about managing technology as a lifecycle discipline. It is structured as an imported-paper example: the original source draft is not included, but the completed GPD artifacts show how a messy existing draft can move through brief, research, outline, draft, fact-check, review, revision, and export.

## Intended Outcome

The reader should understand why technology lifecycle management needs a sponsored pilot, what decisions the pilot should test, and how ownership, evidence, and architecture facilitation should work without creating a new approval bureaucracy.

## Format

- **Type:** Position paper / decision memo
- **Target length:** 1,800-2,500 words
- **Publishing context:** Internal executive review and peer architecture discussion
- **Deadline:** None

## Stakes

The paper addresses hidden run cost, duplicated capability, unclear ownership, delayed modernization, and operational risk in a regulated enterprise context. It must avoid overclaiming regulatory support while still making a concrete strategic recommendation.

## Source Policy

- **Citation style:** Source IDs in research, fact-check, and draft-stage evidence hooks
- **Evidence standard:** Official/public sources for external claims; anonymized author-provided source for proposed operating-model intent
- **Allowed sources:** Web research and anonymized user-provided draft context
- **Source storage:** The private imported source draft is intentionally not included in this example

## Boundaries

- Does not name real organizations, people, employer names, internal titles, or local file paths.
- Does not include the original imported draft.
- Does not choose the final system of record, enterprise taxonomy, pilot domains, or control mapping.

## Key Decisions

| Decision | Rationale | Status |
|----------|-----------|--------|
| Keep the imported source private | The example should demonstrate workflow output without publishing private source material. | Active |
| Use generic regulated-enterprise context | Gives agents enough risk context without exposing a real organization. | Active |
| Treat ownership and four lifecycle decisions as recommendations | Official sources support the need for governed evidence, not the exact operating model. | Active |
